<Subtitle>A binary on/off control whose change takes effect immediately.</Subtitle>

`ekko-switch` is a [Form-Associated Custom Element](/?path=/docs/foundations-forms-face--docs)
that wraps a visually-hidden native `<input type="checkbox">` carrying `role="switch"`. The
input handles keyboard, focus, and `FormData` participation; a styled track and thumb in
Shadow DOM provide the visual on/off state. Per [WAI-ARIA APG](https://www.w3.org/WAI/ARIA/apg/patterns/switch/),
assistive technology announces the control as a switch with "on" / "off" rather than
"checked" / "not checked".

---

## When to use

- Settings that flip a feature on or off **in real time** вҖ” Wi-Fi, Bluetooth, dark mode, notifications
- Per-row toggles in a settings panel where each row independently activates a behaviour
- Live preference toggles that mutate state the moment the user interacts

**Do not use** `ekko-switch` for form fields awaiting submission вҖ” use
[`ekko-checkbox`](/?path=/docs/components-checkbox--docs) when the value is staged until
the user clicks Submit. **Do not use** it to choose between two equivalent options вҖ” use
[`ekko-radio`](/?path=/docs/components-radio--docs) for mutually-exclusive choices.

## Properties & attributes

| Attribute | Property | Type | Default | Description |
| --- | --- | --- | --- | --- |
| `checked` | `checked` | `boolean` | `false` | On/off state. Reflects to/from the inner input. |
| `disabled` | `disabled` | `boolean` | `false` | Disables interaction. Sets `aria-disabled="true"` on the inner input. |
| `name` | `name` | `string` | `''` | Form submission name. |
| `value` | `value` | `string` | `'on'` | Form submission value when on. Matches native checkbox default. |
| `size` | `size` | `'sm' \| 'md' \| 'lg'` | `'md'` | Size scale. |
| `aria-label` | вҖ” | `string` | вҖ” | Accessible name when no slotted label is provided. Forwarded to the inner input. |
| `aria-describedby` | вҖ” | `string` | вҖ” | Describing element id(s). Forwarded to the inner input. |

The `form` getter returns the host form (`HTMLFormElement | null`).

## Methods

| Method | Returns | Description |
| --- | --- | --- |
| `click()` | `void` | Inherited from `HTMLElement`. Programmatic click. Toggles the on/off state and fires `change`. |
| `focus(options?)` | `void` | Inherited from `HTMLElement`. Moves keyboard focus to the host (which retargets to the inner input). |
| `blur()` | `void` | Inherited from `HTMLElement`. Removes focus. |

There is no separate `toggle()` method вҖ” set the `checked` property directly.

---

## Events

| Event | `detail` shape | When fired |
| --- | --- | --- |
| `ekko-change` | `{ checked: boolean, originalEvent: Event }` | When the user toggles the switch via click or keyboard. Not fired when `checked` is set programmatically. |
| `change` | native `Event` | Standard DOM change. Bubbles out of Shadow DOM вҖ” listen here if you don't need the typed detail. |

`ekko-change` `bubbles` and `composes` (crosses the Shadow boundary).

## Accessibility

### Role

The inner `<input>` carries `role="switch"`, so screen readers announce it as a switch
rather than a checkbox. The native `checked` property still drives state вҖ” the role only
changes how the state is verbalised.

### Keyboard

| Key | Action |
| --- | --- |
| `Tab` / `Shift+Tab` | Move focus to/from the switch |
| `Space` | Toggle on/off (native `<input type="checkbox">` behaviour) |
| `Enter` | Toggle on/off on most platforms (browser/AT dependent) |

### Screen reader announcements

| State | Announcement |
| --- | --- |
| Off | "&#123;label&#125;, switch, off" |
| On | "&#123;label&#125;, switch, on" |
| Disabled | "&#123;label&#125;, switch, dimmed" (varies by reader) |

### Focus

The native `<input>` inside the shadow tree is the focus target. The visible track renders
the focus ring via `:focus-visible ~ .track` (sibling combinator) using the
`--ekko-switch-shared-focus-ring-*` tokens. The hidden input is positioned absolutely with a
1Г—1 clip вҖ” it stays in the tab order without affecting layout.

### Labelling

Slotted text is the visible label. If you don't slot a label, set `aria-label` on the host
so the inner input has an accessible name.

```html
<!-- preferred вҖ” visible label -->
<ekko-switch name="wifi">Wi-Fi</ekko-switch>

<!-- no visible label вҖ” use aria-label -->
<ekko-switch name="dark-mode" aria-label="Dark mode"></ekko-switch>
```

## Form integration

```html
<form>
  <ekko-switch name="notifications" checked>Notifications</ekko-switch>
  <ekko-switch name="newsletter">Weekly digest</ekko-switch>
  <ekko-button type="submit">Save preferences</ekko-button>
</form>
```

`FormData` includes `name=value` only when the switch is **on**. When off, the entry is
omitted entirely (matching native `<input type="checkbox">` behaviour).

The `value` attribute defaults to `'on'` to match native semantics. Set it explicitly when
sending a meaningful value (e.g. `value="enabled"`).

`<fieldset disabled>` propagates automatically through `formDisabledCallback` вҖ” every
switch inside becomes disabled and stops contributing to `FormData`. No code change needed.

`form.reset()` restores the initial `checked` attribute via `formResetCallback`. If the
switch was authored with `<ekko-switch checked>`, reset returns it to on; otherwise it
returns it to off.

See [Forms (FACE)](/?path=/docs/foundations-forms-face--docs) for the full form integration story.

## Things to know

### "On / off" is the right mental model

A switch announces and behaves as on/off. Phrase labels as the **thing being toggled**, not
as a verb вҖ” "Notifications" with a switch, not "Enable notifications" with a switch. The
verb is implied by the control type.

### Switches indicate immediate effect

Toggling a switch should mutate state right away. If the user has to click Save afterwards
for the change to apply, you've picked the wrong control вҖ” use a checkbox instead. This is
the single biggest reason teams reach for switches when they shouldn't.

### Programmatic `checked = true` does not fire `ekko-change`

Only user interaction fires the event вҖ” same as native. If you need to react to programmatic
changes (e.g. syncing two switches), wire it up explicitly in your state-update code.

### Announced as a switch, not a checkbox

The component matches the [WAI-ARIA APG Switch pattern](https://www.w3.org/WAI/ARIA/apg/patterns/switch/).
Even though the underlying element is `<input type="checkbox">`, `role="switch"` overrides
how assistive technology verbalises the state. Don't rely on the word "checkbox" appearing
anywhere in the announcement.

### The hidden input is the focus target

If you grab the host's focused state via `:focus-within`, it works. If you read
`document.activeElement`, you'll see the host element (focus retargets at the Shadow
boundary). Use `event.composedPath()` if you need the inner input.
